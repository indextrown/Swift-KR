# 공식 문서 최신화

공식 원문 저장소의 **실제 문서 diff**를 기준으로 이미 번역된 한국어 문서를 최신화한다. 단순히 최신 커밋이 있다는 사실만으로 수정하지 말고, 해당 커밋 또는 머지된 PR이 어느 원문 파일을 어떻게 바꿨는지 확인한 뒤 연결된 한국어 페이지에만 반영한다.

작업 전에 [업스트림 문서 매핑표](references/upstream-registry.md)를 읽는다. 로컬 문서의 첫 부분에 있는 `원문:` 또는 `공식 참고:` 링크가 매핑표보다 더 구체적이면 그 링크를 우선한다.

## 적용 대상 정하기

사용자 요청을 다음 중 하나로 분류한다.

| 요청                   | 대상 선정 방식                                                                             |
| ---------------------- | ------------------------------------------------------------------------------------------ |
| 특정 문서 최신화       | 경로, 제목, 슬러그, 원문 URL로 한국어 문서 하나 또는 관련 문서 묶음을 찾는다.              |
| 특정 PR 또는 커밋 반영 | 해당 PR/커밋이 바꾼 원문 파일을 먼저 찾고, 매핑된 한국어 문서만 대상으로 삼는다.           |
| 전체 최신화            | 매핑표에 등록된 모든 로컬 문서를 후보로 삼고, 업스트림 문서 변경이 있는 페이지만 수정한다. |

경로와 제목이 모호하면 먼저 `rg`로 로컬 문서를 찾는다. 여러 문서가 일치해도 가장 그럴듯한 문서를 임의로 하나만 고르지 말고, 모두 검사한 뒤 실제 원문 변경이 확인된 문서만 갱신한다.

## 원문과 변경 범위 확인하기

1. 로컬 문서의 `원문:`·`공식 참고:` 링크와 매핑표로 정확한 `owner/repo`, 기본 브랜치, 원문 파일 경로를 확정한다. `docs/guide/tca`라는 디렉터리명만으로 모든 원문을 `swift-composable-architecture`로 취급하지 않는다.
2. 공식 저장소의 기본 브랜치 최신 SHA를 조회하고, 원문 파일의 최신 내용을 확인한다. Git 저장소를 내려받아야 하면 작업 트리 밖의 임시 디렉터리에 얕거나 blob-less clone을 만들고, 프로젝트에 외부 저장소를 추가하지 않는다.
3. 기준점은 다음 우선순위로 정한다.
   1. 로컬 문서의 `<!-- upstream-sync: owner/repo@SHA | source: path -->` 주석
   2. 해당 문서의 마지막 로컬 Git 커밋 시각을 기준으로 한 업스트림 변경 창
   3. 기준점이 전혀 없으면 현재 공식 원문과 한국어 문서를 직접 대조하는 최초 점검
4. 기준점부터 최신 기본 브랜치까지 파일 경로를 제한해 커밋과 diff를 확인한다. GitHub PR URL이나 번호가 주어졌다면 `gh pr view`로 머지 여부·merge commit·변경 파일을 확인하고 `gh pr diff`로 내용을 읽는다. `gh`를 쓸 수 없으면 GitHub의 공식 PR/commit 페이지와 Git diff로 같은 정보를 확인한다.
5. 문서 소스(`README.md`, `.md`, `.mdx`, `.docc` article 등)가 아닌 변경, 또는 대상 원문 파일과 무관한 변경은 갱신 근거로 삼지 않는다. 파일 이동·이름 변경은 rename detection으로 이전과 이후 경로를 함께 확인한다.

PR이 아직 머지되지 않았다면 기본적으로 반영하지 않는다. 사용자가 그 PR을 명시적으로 반영해 달라고 했을 때만, "미병합 변경"임을 결과에 밝혀 반영한다.

## 변경을 한국어 문서에 반영하기

1. 각 후보를 `원문 파일 → 관련 PR/커밋 → 한국어 문서 → 반영할 변경`으로 짧게 정리한 뒤 수정한다.
2. 새로 추가된 설명·주의사항·제한·코드 예제·링크·섹션을 빠짐없이 번역한다. 원문에서 제거된 직접 번역 내용은 한국어 문서에서도 제거하거나, 이 저장소 고유의 보충 설명이면 원문과 구분되도록 유지한다.
3. Swift 코드, API 이름, 패키지 식별자, 파일 경로, 명령어, URL은 원문 그대로 유지한다. 코드 변경은 번역으로 추측하지 말고 공식 diff와 일치하게 수정한다.
4. 새 섹션이나 페이지가 생기면 문서 본문을 먼저 추가하고, 그다음 Rspress의 해당 `_meta.json` 또는 내비게이션 파일에 원문 순서를 따라 연결한다. 삭제·이름 변경도 내비게이션과 내부 링크까지 함께 점검한다.
5. 최초로 최신화한 직접 번역 페이지에는 원문 링크 바로 아래에 다음 형식의 HTML 주석을 추가한다. 이후 동기화 때 최신 기본 브랜치 SHA와 실제 원문 경로로 갱신한다.

   ```md
   <!-- upstream-sync: pointfreeco/swift-composable-architecture@<commit-sha> | source: Sources/.../Article.md -->
   ```

   독자에게 보이는 문구는 추가하지 않는다. 여러 공식 저장소를 조합한 자체 가이드는 주석 하나에 억지로 기준점을 합치지 말고, 사용한 원문별로 주석을 나누거나 변경 근거만 결과에 기록한다.

6. 원문 변경이 없거나 한국어 문서가 이미 같은 내용을 반영했다면 파일을 건드리지 않는다. 이미 반영된 변경을 다시 번역하거나 형식만 바꾸는 수정은 하지 않는다.

## 전체 최신화 추가 규칙

전체 최신화에서는 먼저 후보 목록과 근거를 만든 다음, 변경이 확인된 문서만 한 묶음으로 수정한다. 문서마다 다른 업스트림 저장소와 기준점을 유지한다. 한 문서의 업스트림 변경을 같은 디렉터리의 다른 문서에 기계적으로 전파하지 않는다.

변경이 많은 경우에도 다음 순서를 지킨다.

1. 직접 번역 페이지와 공식 원문을 우선 반영한다.
2. 그 변경으로 깨진 목차, `_meta.json`, 상호 링크를 수정한다.
3. `tca-dependencies.md`처럼 이 저장소가 작성한 실전 가이드는 원문에서 직접 근거를 찾을 수 있는 주장·예제만 최신화한다.

## 검증과 결과 보고

수정한 뒤에는 다음을 수행한다.

1. `git diff --check`로 공백 오류를 확인한다.
2. 프로젝트 규칙에 따라 `npm run format`과 `npm run lint`를 실행한다. 포매터가 작업 범위 밖 파일을 바꾸면 그 변경을 분리하고, 원래 있던 사용자 변경은 되돌리거나 덮어쓰지 않는다.
3. 수정한 코드 블록은 공식 diff와 다시 대조하고, 새 문서와 `_meta.json`의 경로·순서를 확인한다.

결과에는 한국어 문서별로 원문 저장소와 경로, 반영한 PR/커밋, 핵심 변경, 검증 결과를 간단히 적는다. 변경할 내용이 없으면 검사한 범위와 "최신화할 공식 문서 변경 없음"을 명확히 보고한다. 커밋·푸시는 사용자가 별도로 요청할 때만 한다.

## TCA 예시

- PR #3954는 `TreeBasedNavigation.md`의 테스트 예제 reducer를 수정했다. 따라서 `docs/guide/tca/tree-based-navigation.md`만 대조·갱신 대상으로 삼는다.
- PR #3955는 `StackBasedNavigation.md`의 path mutation과 테스트 예제를 수정했다. 따라서 `docs/guide/tca/stack-based-navigation.md`만 대조·갱신 대상으로 삼는다.

이처럼 PR 제목이 아니라 **Files changed와 실제 diff**로 대상 한국어 페이지를 결정한다.
