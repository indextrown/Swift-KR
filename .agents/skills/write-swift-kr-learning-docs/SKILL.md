# Swift-KR 학습 문서 작성

기본 Swift 문법을 아는 입문자가 낯선 개념을 처음부터 따라갈 수 있는 한국어 학습 문서를 만든다. 문서의 기준 골격과 작성 검사는 [문서 포맷](references/document-format.md)을 따른다.

메모리 영역, ARC, 객체 그래프처럼 소유 관계를 그림으로 설명할 때는 [메모리 다이어그램 템플릿](references/memory-diagram-template.md)을 읽고 JSON 기반 SVG 생성기를 사용한다.

## 작업 순서

### 1. 저장소와 문서 위치 확인

1. 저장소의 `AGENTS.md`, `rspress.config.ts`, `docs/_nav.json`, 대상 폴더의 `_meta.json`을 읽는다.
2. 같은 섹션의 기존 문서에서 제목, 말투, 코드 스타일, 내비게이션 순서를 확인한다.
3. 새 주제의 학습 목표와 독자 수준을 한 문장으로 정한다. 사용자가 별도로 지정하지 않으면 기본 Swift 문법은 알지만 주제는 처음 접하는 독자를 대상으로 한다.
4. 현재 형식의 기준 사례가 필요하면 `docs/guide/design-patterns/dependency-injection.md`를 읽는다. 내용이 아니라 설명 순서와 난이도만 참고한다.

### 2. 신뢰할 수 있는 자료 조사

1. 새 기술 내용을 작성하거나 최신성이 필요한 내용을 고칠 때는 인터넷 자료를 조사한다.
2. Swift 문법과 플랫폼 API는 Swift·Apple 공식 문서를 우선하고, 라이브러리는 공식 저장소와 공식 문서를 우선한다.
3. 디자인 원칙과 패턴은 원저자의 글, 논문, 서적의 공식 공개 자료처럼 개념의 근거가 되는 출처를 찾는다.
4. 검색 결과 요약만으로 쓰지 말고 실제 원문을 읽는다. 서로 다른 출처로 핵심 정의와 Swift 적용 방식을 교차 확인한다.
5. 출처가 설명하는 사실과 작성자가 도출한 권장 사항을 구분한다. 중요한 주장 가까이에 링크를 달고 문서 끝 `참고 자료`에 다시 정리한다.

단순한 문장 다듬기나 구조 변경처럼 새로운 사실을 추가하지 않는 작업에서는 불필요한 재조사를 생략한다.

### 3. 선행 개념부터 설계

1. 본문에 사용할 약어, 설계 용어, Swift 고유 문법, 프레임워크와 플랫폼 이름을 먼저 목록으로 만든다.
2. 독자가 모를 수 있는 용어는 최초 사용 전에 쉬운 말로 정의한다. 면접 한 줄 요약에 등장하는 용어도 예외로 두지 않는다.
3. 요약, 도입부, 첫 예제를 이해하는 데 필요한 용어는 `먼저 알아둘 용어`에서 설명한다. 뒤쪽에서 처음 등장하는 용어는 해당 문단에서 바로 정의해 초반 표가 지나치게 커지지 않게 한다.
4. 여러 선행 용어가 필요하면 표를 사용한다. 용어가 하나뿐이면 짧은 문단으로 설명한다.
5. `UIKit`, `SwiftUI`, `Combine`처럼 익숙하다고 가정하기 쉬운 프레임워크 이름도 대상 독자가 모를 수 있다면 역할을 한 문장으로 설명한다.
6. SOLID처럼 여러 원칙을 묶은 약어는 전체 이름과 각 구성 요소를 먼저 소개하고, 이 문서에서 실제로 다룰 범위를 밝힌다.
7. 전문 용어를 다른 전문 용어만으로 정의하지 않는다. 일상어 설명 뒤에 정확한 기술 용어를 연결한다.

### 4. 문제에서 해결로 설명

1. 문서 상단에 `면접 답변 한 줄 요약`을 배치한다. 정의, 핵심 작동 방식, 얻는 효과를 한 문장에 담는다.
2. 독자가 겪을 만한 작은 문제나 가장 단순한 Swift 코드에서 시작한다.
3. 기존 방식의 문제를 코드 아래에서 구체적으로 설명한 뒤 개선된 코드를 보여준다.
4. 각 코드 블록 앞에는 확인할 관점을, 뒤에는 코드가 바꾼 점을 설명한다.
5. 비슷한 개념은 역할이나 답하는 질문을 기준으로 표에서 비교한다.
6. 장점만 나열하지 말고 비용, 오용, 사용하지 않아도 되는 경우를 함께 설명한다.
7. 마지막에는 적용 순서 또는 판단 체크리스트와 짧은 면접 후속 질문을 제공한다.

주제에 맞는 정확한 섹션 선택과 예시는 [문서 포맷](references/document-format.md)을 따른다. 선택 섹션은 학습에 도움이 될 때만 추가한다.

### 5. Swift 예제 작성

1. 예제 하나에는 한 가지 학습 포인트만 둔다.
2. 같은 도메인과 타입 이름을 문서 전체에서 유지해 독자가 개념 변화에만 집중하게 한다.
3. 생략한 프로덕션 로직은 주석으로 명시하고, 핵심 문법은 실행 가능한 형태로 작성한다.
4. 프로토콜, 클로저, 제네릭, `any`, 동시성처럼 선택지가 있다면 왜 해당 표현을 골랐는지 설명한다.
5. 테스트 가능성이 주제와 관련되면 실제 구현과 테스트 대역을 함께 보여주고 결과가 안정되는 이유를 설명한다.
6. 컴파일이나 테스트가 가능한 프로젝트 환경이 있으면 예제를 검증한다. 그렇지 않으면 API 이름, 격리 규칙, 비동기 시그니처를 공식 문서와 대조한다.

### 6. Rspress에 연결

1. 새 페이지에 한국어 `title`과 50~160자의 구체적인 `description` frontmatter를 작성한다.
2. 페이지 파일을 주제에 맞는 폴더에 두고 해당 `_meta.json`에 slug와 표시 이름을 추가한다.
3. 새 최상위 섹션이거나 상단에서 바로 찾아야 하는 경우에만 `docs/_nav.json`을 수정한다.
4. 관련 문서와 자연스럽게 이어질 필요가 있을 때만 내부 링크를 추가한다.
5. 기존 순서와 사용자 변경을 보존하고 관련 없는 파일은 수정하지 않는다.

### 7. 교육 자료 품질 검사

다음 질문에 모두 답한 뒤 완료한다.

- 제목과 한 줄 요약만 읽어도 개념의 목적을 말할 수 있는가?
- 낯선 약어, 전문 용어, 프레임워크 이름이 설명보다 먼저 등장하지 않는가?
- 첫 예제가 해결하려는 문제가 구체적인가?
- 개선 전후 코드에서 바뀐 이유가 설명되어 있는가?
- 유사 개념의 경계와 흔한 오해를 구분했는가?
- 언제 사용하고 언제 사용하지 않을지 판단할 수 있는가?
- 코드와 사실을 공식 또는 원전 자료로 확인했는가?
- 같은 한국어 용어와 `-요.` 문체를 일관되게 사용했는가?

### 8. 검증

1. `git diff --check`를 실행한다.
2. `npm run format`과 `npm run lint`를 실행한다.
3. `npm run build`로 정적 페이지 생성을 검증한다.
4. 포매터가 관련 없는 파일을 바꿨다면 사용자 변경은 보존하고 도구가 만든 무관한 변경만 되돌린다.
5. 완료 보고에는 작성한 문서, 사용한 핵심 출처, 내비게이션 변경, 검증 결과를 간단히 적는다.
