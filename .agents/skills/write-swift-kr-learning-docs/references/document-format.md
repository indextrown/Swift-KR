# Swift-KR 학습 문서 포맷

새 문서를 만들거나 기존 문서를 전면 개정할 때 이 구조를 기준으로 삼는다. 모든 선택 섹션을 억지로 채우지 말고 주제를 이해하는 데 필요한 것만 사용한다.

## 필수 골격

```markdown
---
title: Swift로 이해하는 {주제}
description: { 대상 기술과 독자가 얻게 될 구체적인 내용을 50~160자로 작성 }
---

# Swift로 이해하는 {주제}

> **면접 답변 한 줄 요약:** {정의 + 핵심 작동 방식 + 효과를 한 문장으로 작성}

{독자가 이 개념을 배워야 하는 이유와 문서의 범위}

## 먼저 알아둘 {설계/언어/라이브러리} 용어

| 용어   | 쉬운 뜻                                      |

{가장 작은 예제와 설명}

{개선 전 Swift 코드와 구체적인 문제}
```

## 섹션별 규칙

### 면접 답변 한 줄 요약

- 첫 문장을 정의로 시작한다.
- “무엇을 어떻게 해서 무엇이 좋아지는지”를 담는다.
- 줄이 길어지더라도 설명하지 않은 전문 용어로 억지로 줄이지 않는다.
- 본문의 결론과 다른 과장된 장점을 넣지 않는다.

### 선행 용어

- 상단 요약, 도입부, 첫 코드 설명에 등장하는 용어부터 검사한다.
- 독자가 기본 Swift 문법만 안다고 가정한다.
- 약어는 영문 전체 이름과 한국어 이름을 함께 적는다.
- `SOLID`, `IoC`, `DIP`, `ARC`, `ABI`, `Sendable`, existential처럼 검색이 필요한 단어는 최초 등장 전에 설명한다.
- `UIKit`, `SwiftUI`, `Combine` 같은 프레임워크와 플랫폼 이름도 독자가 역할을 모를 수 있다면 짧게 정의한다.
- 용어가 둘 이상이면 표를 사용하고, 한 개면 문단으로 풀어 쓴다.
- 이후 본문에서 더 자세히 다루더라도 초반에는 길을 잃지 않을 정도의 정의를 먼저 준다.
- 뒤쪽 섹션에서만 필요한 용어는 해당 섹션의 최초 등장 지점에서 정의해 초반 용어표가 백과사전처럼 커지지 않게 한다.

### 예제 흐름

하나의 도메인을 유지하며 다음 순서로 전개한다.

1. 가장 단순하지만 문제가 있는 코드
2. 실행·테스트·변경 시 드러나는 구체적인 불편
3. 최소한의 개선
4. 필요한 경우에만 추상화나 추가 패턴 도입
5. 테스트 또는 호출 코드에서 달라진 결과

완성된 구조만 먼저 보여주지 않는다. 독자가 각 단계가 필요한 이유를 확인하게 한다.

### 비교와 시각화

- 두 개념은 짧은 문단이나 목록으로 충분하면 표를 만들지 않는다.
- 세 개 이상의 용어, 반복되는 비교 기준, 의존 방향처럼 관계가 중요하면 표나 작은 텍스트 다이어그램을 사용한다.
- 코드·데이터·힙·스택 영역이나 ARC 소유 관계를 설명하는 SVG는 [메모리 다이어그램 템플릿](memory-diagram-template.md)을 사용해 같은 시각 문법을 유지한다.
- 장식 목적의 다이어그램은 추가하지 않는다.

### 적용 기준

- “항상”, “무조건”, “정답” 같은 표현을 근거 없이 사용하지 않는다.
- 장점과 함께 타입 증가, 탐색 비용, 런타임 오류 가능성 같은 비용도 설명한다.
- 단순한 코드에는 적용하지 않아도 되는 조건을 명시한다.
- 패턴 이름보다 해결하려는 변화와 경계를 중심으로 권장한다.

### 면접 후속 질문

- 본문 핵심에서 3~5개를 고른다.
- 정의 차이, 선택 이유, 장단점, 대안처럼 실제로 이어질 질문을 사용한다.
- 답변은 결론부터 말하고 근거를 덧붙인다.

### 참고 자료

- Swift와 Apple API는 공식 문서를 우선한다.
- 라이브러리는 공식 저장소와 공식 문서를 우선한다.
- 디자인 패턴과 원칙은 원저자 또는 원전 자료를 포함한다.
- 블로그는 공식 자료만으로 설명하기 어려운 Swift 실전 사례를 보완할 때 사용한다.
- 본문에서 실제로 확인한 자료만 목록에 넣는다.
